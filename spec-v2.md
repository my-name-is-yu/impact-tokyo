## セクション0: ビジョン & 課題

### ワンライナー

**「介護者がケアだけに集中できるよう、それ以外の全て — 記録、報告、手続き、情報収集 — をAIオーケストレーションが自律的に一気通貫で処理する」**

---

### 課題の規模

| 指標 | 数値 |
|------|------|
| 介護離職（年間） | 約9.3万人 |
| ビジネスケアラー | 約365万人 |
| 家族介護者のうつ傾向 | 約3割 |
| 東京の高齢化率 | 23%超（2026年時点） |

介護の現場で失われているのは「時間」だけではない。家族介護者は本来のケア以外の業務 — 記録、報告書作成、制度調査、書類準備 — に膨大なエネルギーを消耗している。その結果として離職・燃え尽きが起きている。

---

### 既存ソリューションの限界

| ソリューション | 限界 |
|--------------|------|
| CareNote、ケアコラボ等の介護記録アプリ | 「入力補助ツール」止まり。AIが能動的に動くエージェント機能がない |
| 一般的なAIチャット（ChatGPT等） | 介護特化の知識・文脈がない。記録との連携がない |
| 行政の窓口・ポータル | 情報が散在。手続きを一気通貫でガイドする機能がない |

**MimamoAIの差別化:** 既存アプリが「記録ツール」であるのに対し、MimamoAIは「AIオーケストレーション」として能動的に動く。記録構造化・異常検知・報告書生成・手続きガイドという複数のAI処理をオーケストレーター（指揮者）が連携・統合し、ユーザーが操作しなくてもワークフロー全体が自律実行される。ユーザーは確認して承認するだけでいい。

---

### AIオーケストレーション vs AIツール（差別化の核心）

| 観点 | AIツール（既存アプリ） | AIオーケストレーション（MimamoAI） |
|------|---------------------|----------------------------------|
| 起点 | ユーザーが操作する | AIが先回りして提案・実行する |
| ユーザーの役割 | 入力 → 結果を受け取る | 確認して承認するだけ |
| 記録後の動き | 記録が保存されて終わり | 「食欲が3日連続低下。ケアマネに報告しますか?」と能動アラート |
| 報告書 | ユーザーが「生成」ボタンを押す | 月末にAIが「報告書ドラフトを作成しました。確認しますか?」と通知 |
| 手続き | ユーザーが検索・質問する | 更新時期が来たらAIが先回りして「申請書を準備しました」と通知 |
| 処理構造 | 単一のAIが単発で応答する | 記録→分析→報告→手続きをオーケストレーター（n8n）が連携・統合して自律実行 |

**エージェント動作の具体例:**
1. 記録入力後 → AIが「食欲が3日連続で低下しています。ケアマネジャーに報告しますか?」と能動アラート
2. アラート承認 → AIがケアマネへの連絡文を自動起草、ユーザーは確認して送信するだけ
3. 月末 → AIが記録を集約して報告書ドラフトを自動生成、通知で知らせる
4. 更新時期 → AIが先回りして「要介護認定の更新時期です。申請書類を準備しました」と通知

### P0-A: ケア記録の自動構造化

**概要:** 自然言語テキスト（および写真）を入力すると、AIが標準化されたケア日誌フォーマットに構造化する。「ケア以外の作業をAIが代行する」の起点となる機能。

#### 入力
- 自由テキスト（例: 「今日は朝から食欲なくて、昼もほとんど食べなかった。3回トイレに行って、夕方に少し混乱があった」）
- オプション: 写真（傷・むくみ・食事の様子など。最大5枚）
- 記録日時（自動補完、手動修正可）
- 記録者（ログイン中のユーザー）
- 被介護者（ハードコード、デモ用）

#### 処理（Claude API）
- **プロンプト方針:**
  - 入力テキストから以下の項目を抽出・補完: 食事（量・内容）、排泄（回数・性状）、睡眠、服薬、体調（痛み・発熱・むくみ等）、精神状態（穏やか・不安・混乱等）、転倒リスク、特記事項
  - 写真が添付されている場合: 画像の内容（傷の状態、食事の量感等）をテキスト補足として追加
  - 不明・未記載の項目は「未記録」とし、推測で埋めない
  - 出力はJSON構造で返す（フロントエンドが表示に利用）
  - トーン: 医療・介護の専門用語を適度に使用しつつ、家族介護者でも読みやすい表現

**プロンプトテンプレート:**
```
あなたは介護記録の構造化アシスタントです。
以下の自由記述から、標準的なケア日誌の各項目を抽出してください。
推測や補完はせず、記述から読み取れる情報のみを記録してください。

入力テキスト: {user_input}
添付画像の説明: {image_description_or_none}

以下のJSONフォーマットで出力してください:
{
  "date": "YYYY-MM-DD",
  "meal": { "breakfast": "", "lunch": "", "dinner": "", "hydration": "" },
  "elimination": { "urine_count": null, "stool": "" },
  "sleep": { "hours": null, "quality": "" },
  "medication": { "taken": null, "notes": "" },
  "physical": { "temperature": null, "pain": "", "edema": "", "other": "" },
  "mental": { "mood": "", "cognition": "", "behavior": "" },
  "fall_risk": "",
  "special_notes": "",
  "care_given": []
}
```

#### 出力
- 構造化されたケア日誌（編集可能なフォームとして表示）
- 確認・修正後に保存するボタン
- 保存後、記録一覧に追加
- **保存時に異常パターン検知ワークフロー（P2）をトリガー**

#### エッジケース
| ケース | 対応 |
|--------|------|
| 入力が極端に短い（例: 「普通」） | 「情報が少ないため詳細を確認してください」と表示し、追加入力を促すフォームを出す |
| 写真が鮮明でない・関係ない画像 | 画像からの情報は「判別不可」として記録し、テキスト情報のみで構造化 |
| 複数日分をまとめて入力 | 最新日として記録し、「複数日のデータが含まれている可能性があります。ご確認ください」と警告 |
| ネットワークエラー | 入力テキストをローカルストレージに一時保存し、再接続後に再送信 |

### P0-B: 介護手続きガイド + 書類生成

**概要:** 対話形式で介護手続き（介護保険申請、要介護度更新、各種給付・手当等）をガイドし、必要書類のドラフトを生成する。「AIがケア以外を代わりにやる」ビジョンの最も分かりやすい証拠。

#### 入力
- 自由質問テキスト（例: 「父の介護度を上げるにはどうすればいいですか?」）
- 状況のクイック選択（任意: 「新規申請」「更新」「異議申し立て」「制度調査」）
- 都道府県・市区町村（手続き窓口情報の精度向上のため）

#### 処理（Claude API）
- **Step 1 (分類):** 質問を手続きカテゴリに分類
  - 介護保険関連（新規申請・更新・区分変更・異議申し立て）
  - 給付・助成関連（高額介護サービス費・住宅改修・福祉用具）
  - 職場制度関連（介護休業・介護休暇・所定外労働制限）
  - 医療連携関連（主治医意見書・医療費控除）
- **Step 2 (ガイド生成):** ステップバイステップの手続き説明を生成
  - 必要書類リスト
  - 申請窓口（市区町村の担当部署）
  - 所要期間の目安
  - よくある落とし穴
- **Step 3 (書類ドラフト):** ユーザーが「書類を作成する」を選択した場合、必要情報を収集し書類ドラフトを生成

**プロンプトテンプレート:**
```
あなたは介護手続きのエキスパートアシスタントです。
日本の介護保険制度・社会保障制度に精通しており、家族介護者が迷わず手続きを進められるよう支援します。

ユーザーの状況: {user_query}
都道府県: {prefecture}

以下の形式で回答してください:
1. 手続きの概要（何のための手続きか）
2. ステップ一覧（番号付き、各ステップ1〜2行）
3. 必要書類リスト
4. 申請窓口
5. 注意点・よくある失敗

不明な制度や地域固有の詳細は「お住まいの市区町村にご確認ください」と明記し、推測で回答しないこと。
```

#### 出力
- チャット形式のガイド表示
- 手続きステップのチェックリスト（完了マーク付き）
- 生成された書類ドラフト（編集可能テキストエリア）
- 書類のテキストコピーまたはPDFダウンロード

#### エッジケース
| ケース | 対応 |
|--------|------|
| 制度が地域によって異なる | 「詳細はお住まいの市区町村へご確認ください」を明記 |
| 法改正で情報が古い可能性 | 回答末尾に「情報は {knowledge_cutoff} 時点のものです。最新情報はご確認ください」を付記 |
| 医療・法律的判断が必要な質問 | 「専門家（ケアマネジャー・弁護士・医師）にご相談ください」と誘導 |
| 質問の意図が不明 | 状況を確認するクラリファイ質問を返す |

### P1: ケアマネ向け報告書PDF生成

**概要:** 一定期間のケア記録を集計・要約し、ケアマネジャーへの月次報告書をPDFで生成する。デモではサンプルデータで動かす。月末にAIが能動的に「ドラフトを作成しました」と通知するフローを見せることが重要。

#### 入力
- 報告期間（デフォルト: 前月1日〜末日、手動変更可）
- 被介護者情報（登録済みから自動取得 / デモではハードコード）
- 報告書テンプレート選択（標準 / 簡易）
- 追記事項（任意の自由テキスト）

#### 処理（Claude API + PDF生成）
- **Step 1 (Claude API):** 指定期間のケア記録（JSON配列）を渡し、以下を生成
  - 期間サマリー（体調傾向・特記事項のハイライト）
  - 項目別集計テキスト（食事状況、排泄状況、精神状態の変化傾向）
  - 医療機関への申し送り候補（続く症状・異常値）
  - 今後のケアプランへの提案（オプション）
- **Step 2 (PDF生成):** サマリー + 集計データをHTMLテンプレートに流し込み、`window.print()` でPDF化

**プロンプトテンプレート:**
```
あなたは介護支援専門員（ケアマネジャー）への月次報告書作成アシスタントです。
以下の {N} 件のケア記録を読み、専門的かつ簡潔な月次報告書を作成してください。

被介護者: {care_recipient_name}（{age}歳、要介護{level}）
報告期間: {start_date} 〜 {end_date}
記録件数: {count}件

ケア記録データ:
{json_records}

以下のセクションを含むレポートを作成してください:
1. 総合所見（200字以内）
2. 体調・ADL（食事・排泄・睡眠・移動）の傾向
3. 認知・精神状態の変化
4. 特記事項・インシデント
5. 次月への申し送り事項
```

#### 出力
- プレビュー画面（HTMLレンダリング）
- 修正可能なテキストフィールド（各セクション）
- 「PDFダウンロード」ボタン
- 「コピー（テキスト）」ボタン

#### エッジケース
| ケース | 対応 |
|--------|------|
| 指定期間の記録が0件 | 「この期間に記録がありません」と表示し、期間変更を促す |
| 記録が3件以下で少ない | 「記録件数が少ないため、報告書の精度が低い可能性があります」と警告表示 |
| PDF生成失敗 | テキスト形式でのコピー・ダウンロードにフォールバック |

### P2: 異常パターン検知アラート（n8n）

**概要:** ケア記録保存時に直近7日間の記録をAIで分析し、医療的注意が必要なパターンを自動検知してアラートを出す。「AIが先回りして動く」エージェント動作の中核。

- 新規記録保存時にn8nワークフローをトリガー
- 直近7日間の記録をClaude APIで分析
- 異常検知時: プッシュ通知 + ダッシュボードの「AIからの気づき」に表示
- ユーザーが承認 → AIがケアマネへの連絡文を自動起草

## セクション5: APIフロー

### 5.1 Claude API 連携

#### A. ケア記録構造化（P0-A）

```
[フロントエンド]
     |
     | POST /api/records/structure
     | body: { raw_text, images_base64[], care_recipient_id }
     |
     v
[Cloud Functions for Firebase]
     |
     |-- 画像が添付されている場合:
     |   1. 各画像を base64 → Claude Vision に渡し、画像の説明テキストを生成
     |      (claude-3-5-sonnet, messages API, vision対応)
     |   2. 画像説明テキストを raw_text に付記
     |
     |-- Claude API 呼び出し (claude-3-5-haiku)
     |   messages: [
     |     { role: "system", content: <構造化システムプロンプト> },
     |     { role: "user", content: raw_text + image_descriptions }
     |   ]
     |   response_format: JSON object
     |
     v
[Cloud Functions for Firebase]
     |-- JSONレスポンスをパース・バリデーション
     |-- Firestore の CareRecord ドキュメントに status=draft で保存
     |-- n8n Webhook をトリガー（P2: 異常パターン検知）
     |
     v
[フロントエンド]
     |-- 構造化結果をフォームに展開・表示
     |-- ユーザーが確認・修正 → 「保存」 → PATCH /api/records/:id { status: "saved" }
```

**Claude API パラメータ:**
- Model: `claude-3-5-haiku-20241022`（コスト最適化）
- Max tokens: 1000
- Temperature: 0（確定的な構造化）

#### ワークフロー 1: 異常パターン検知アラート（P2 — 重点）


### 5.3 技術スタック

| レイヤー | 採用技術 | 理由 |
|---------|---------|------|
| フロントエンド | Lovable + Firebase Hosting | UI高速生成（Lovable）、フロントエンド配信（Firebase Hosting） |
| DB | GCP Cloud Firestore | NoSQL・無料枠あり・ネストしたドキュメント構造でJSONB相当のデータを格納 |
| Auth | Firebase Authentication | Google OAuth対応・無料 |
| バックエンド | Cloud Functions for Firebase | Claude API呼び出し等のサーバーサイド処理 |
| AI（記録構造化） | claude-3-5-haiku-20241022 | コスト最適化 |
| AI（報告書・ガイド） | claude-3-5-sonnet-20241022 | 品質重視 |
| ワークフロー自動化 | n8n | AIオーケストレーションレイヤー（異常検知・通知・リマインダー等のワークフロー連携） |
| ストレージ（写真） | Firebase Storage | 無料枠あり・Firebase エコシステムで完結 |
| PDF生成 | window.print() + 印刷用CSS | Lovableとの相性が最も良い |

**技術スタックの統一方針:** Lovable（フロントエンドUI生成）+ Firebase/GCP（DB・Auth・バックエンド）+ n8n（AIオーケストレーション）で統一する。

### 5.4 PDF生成の方法

**採用方式: ブラウザ側 `window.print()` + 印刷専用CSS（Lovable との相性が最も良い）**

```
手順:
1. 報告書データを HTML テンプレートコンポーネントに渡してレンダリング
2. 印刷専用 CSS (@media print) で:
   - ヘッダー/フッター/ナビゲーションを非表示
   - A4サイズにフォーマット
   - ページ区切りの制御 (page-break-before/after)
3. JavaScript: window.print() 呼び出し → ブラウザのPDF保存ダイアログ
```

### 5.5 写真のハンドリング

```
アップロードフロー:
1. フロントエンド: <input type="file" accept="image/*"> で最大5ファイル選択
2. クライアント側リサイズ: Canvas API で長辺1024px以下にリサイズ（Claude API 入力コスト削減）
3. バックエンド:
   a. Firebase Storage にアップロード
   b. 返却された URL を Firestore の CareRecord.photos フィールドに保存
4. Claude Vision 呼び出し時:
   - 保存済みURLから画像を取得 または base64 を直接渡す
   - image/jpeg で送信（png より軽量）

セキュリティ:
- 署名付きURL（有効期限付き）でアクセス制御
- ファイルタイプ検証（MIME type: image/jpeg, image/png, image/webp のみ）
- 最大ファイルサイズ: 5MB/枚（リサイズ後）
```
